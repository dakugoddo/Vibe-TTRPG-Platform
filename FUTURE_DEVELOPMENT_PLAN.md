## 1. Мультиплеер и Сетевая Архитектура (Оптимизированная LAN-модель)

На данный момент сетевая игра планируется не через выделенные WAN-серверы, а через эмуляцию локальной сети (LAN).

### 1.1 Архитектурный Пивот: y-websocket вместо y-webrtc
**Критика исходного решения:** `y-webrtc` (P2P mesh сеть) создает O(N^2) соединений и требует отдельного сигнального сервера. В условиях виртуальной LAN это избыточно, нестабильно и ресурсоемко. 
**Решение:**
- Игрок 1 (Хост / GM) локально запускает **и Express-сервер, и WebSocket-сервер (`y-websocket`)**.
- Остальные игроки переходят по IP-адресу Хоста в LAN (например, `http://192.168.1.55:5173`).
- **Топология "Звезда"**: Все клиенты синхронизируются только с Хостом. Огромная экономия оперативной памяти и сетевого трафика на клиентах, максимально низкий пинг, стабильность при любых разрывах.

### 1.2 Оптимизация Yjs (Предотвращение разрастания ОЗУ)
**Критика:** CRDT структуры (Yjs) бесконечно сохраняют историю действий (CRDT Tombstones). Интерактивное позиционирование (рисование, drag-move) за пару часов сессии "съест" гигабайты памяти браузера.
**Решения (Обязательно к реализации):**
1. **Sub-documents (Разделение Документов):** Глобальный Y.Doc (База данных) должен быть отделен от Канвасов. Каждый игровой Канвас открывается как отдельная WebSocket-комната (или Yjs Sub-document). Игрок получает события только того Канваса, где находится.
2. **Yjs State Compaction:** Внедрение механизма "очистки". Так как источник истины — это `.md` файлы на жестком диске, сервер должен при старте (или перезагрузке комнаты) строить свежий 'чистый' Y.Doc без истории и раздавать его клиентам, очищая браузерную память.
3. **Awareness Throttling:** Эфемерные данные мультиплеера (Курсоры-указки, селекшн-боксы) передаются через `Yjs Awareness Protocol` (не попадают в историю Y.Doc). Это нужно ограничить (throttle) до 20-30 кадров в секунду, чтобы не забивать LAN-канал.

### 1.3 Ассеты (Изображения, Музыка, Токены)
- **Хранение**: Все файлы (картинки, MP3) хранятся **только у Хоста** в `world/assets/`.
- **Авто-Оптимизация (WebP)**: При загрузке картинки в игру, Express-сервер обязан сжимать её в формат `.webp` (через sharp) до сохранения. Это уменьшит трафик на 60-80% при скачивании клиентами.
- **Подгрузка и Кеш (Lazy Loading)**: Клиент запрашивает ассеты по HTTP. Сохраняет их в CacheStorage/IndexedDB браузера. При следующем входе игра загружается мгновенно из локального кеша.

## 2. Развитие Интерфейса и Игроков

### 2.1 Ролевая модель (GM / Player)
- При входе выбирается роль и **Имя Игрока**.
- Формируется папка инвентаря `users/{PlayerName}/`.
- Хост (GM) имеет полный доступ (`gm/`, `users/`).
- Игроки получают права на запись только в рамках своих папок/сущностей (на уровне UI валидации или серверной проверки WebSocket сообщений).

### 2.2 Инструменты Гейммастера (Канвас)
- **Туман войны (Fog of War):** Два слоя на канвасе — маска (скрывает) и прозрение (открывает).
- **Сетка и Привязка (Snap-to-Grid):** Сетка с настройкой шага (квадраты, гексы). Инструмент примагничивания токенов.
- **Пинг / Мультиплеерная Указка:** Зажатие горячей клавиши для создания пульсирующего пинга на карте у всех игроков (через Yjs Awareness).

### 2.3 Чат и Дайсы
- Интеграция бросков костей (`/r 1d20+5`) с распаршиванием параметров из математического движка.
- Вывод логов изменения хитпоинтов и статусов (кто кого ударил) в общий чат.

### 2.4 Локальный Undo/Redo и Режим редактирования
- Интеграция нативного `Y.UndoManager` из Yjs для реализации Selective Undo (отменяются только локальные операции игрока, чужие сохраняются). 
- Переход в порталы и смена канвасов отслеживает флаг `isDirty`. Если пользователь вносил изменения, при уходе с холста его должны уведомить: *"Переход очистит вашу персональную историю отмен (Undo) для этого холста"*.

## 3. Этапы для следующих сессий (Приоритет: Оптимизация)

1. **Фаза Сетевого Фундамента:** 
   - Замена `y-webrtc` на локальный WebSocket сервер на стороне Express.
   - Разделение глобального состояния на Sub-documents (отдельно База, отдельно конкретные Канвасы).
2. **Фаза Ассетов и Кеша:** 
   - Авто-конвертация изображений в WebP.
   - Настройка Cache API для HTTP `GET /api/assets/`.
3. **Фаза Интерактивного Канваса (Performance):** 
   - Внедрение Throttling'а на операции Drag/Move.
   - Реализация мультиплеерных эфемерных курсоров (Awareness).
   - Снаппинг к сетке (Grid).
4. **Фаза Ролей и Доступа:** 
   - Разграничение инвентарей.
   - Механизм выдачи прав на токены героям.
