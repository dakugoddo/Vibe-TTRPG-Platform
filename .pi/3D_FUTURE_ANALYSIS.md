# 🧊 3D-БУДУЩЕЕ: Анализ возможности интеграции 3D-движка

> Дата: 2026-05-02
> Вопрос: Можно ли потом прикрутить 3D как Tabletop Simulator? Имеет ли Tauri+Rust преимущества?

---

## 🎯 КОНТЕКСТ ВОПРОСА

Пользователь хочет:
1. **Сейчас**: лёгкое приложение для текста, изображений, музыки, видео, 2D-канвасов
2. **Потом (опционально)**: 3D-пространство с шейдингом и физикой как Tabletop Simulator
3. Вопрос: стоит ли сразу закладывать Tauri+Rust чтобы потом 3D вкрутить?

---

## 🔬 ВАРИАНТ 1: Остаться на React+TS → 3D через Three.js в WebView

### Как это работает
Three.js (или Babylon.js) рендерит 3D-сцену в `<canvas>` элемент. WebGPU (уже в Chrome 113+) даёт нативный доступ к GPU.

### Что можно сделать
- ✅ 3D-стол с миниатюрами, кубами, костями
- ✅ Шейдеры (GLSL/ WGSL), освещение, тени
- ✅ Физика через cannon.js / rapier (WebAssembly)
- ✅ Импорт .glb/.gltf моделей из Steam Workshop
- ✅ Виртуальный рабочий стол с 3D-панорамой

### Ограничения
- 🟡 Производительность: WebGL медленнее нативного на ~20-30% (но WebGPU сокращает разрыв до ~5-10%)
- 🟡 Нет доступа к некоторым фичам GPU (ray-tracing, DLSS)
- 🟡 Большие 3D-сцены (миллионы полигонов) будут тормозить
- 🟡 Нет возможности рендерить в отдельном окне без хаков
- 🔴 Мобильные браузеры: WebGL ограничен, WebGPU почти нет

### Вердикт: **ВПОЛНЕ ДОСТАТОЧНО** для 90% TTRPG-юзкейсов
Tabletop Simulator в браузере через Three.js — реально. Примеры: `boardgame.io` + Three.js, `Tabletopia`.

## 🔧 ВАРИАНТ 2: Tauri + Rust

### Что даёт Tauri поверх текущего стека

```
Сейчас:                    Tauri:
┌─────────────┐           ┌─────────────────┐
│  Браузер     │           │  Tauri WebView   │ ← тот же HTML/CSS/TS!
│  (Chrome)    │           │  (системный)     │
└─────────────┘           └─────────────────┘
       │                          │
┌─────────────┐           ┌─────────────────┐
│  Express     │           │  Rust Backend    │ ← заменяет Express
│  (Node.js)   │           │  (быстрее,       │
│              │           │   безопаснее)    │
└─────────────┘           └─────────────────┘
       │                          │
┌─────────────┐           ┌─────────────────┐
│  Файловая    │           │  Нативный        │ ← полный доступ
│  система     │           │  файловый API    │    к диску
│  (через API) │           │                  │
└─────────────┘           └─────────────────┘
                                  │
                           ┌─────────────────┐
                           │  • Steam SDK     │ ← можно дёргать
                           │  • Системные     │    Steam API
                           │    уведомления   │
                           │  • Нативный      │
                           │    WebSocket     │
                           └─────────────────┘
```

### Преимущества Tauri для 3D

| Фича | Как в Tauri |
|------|------------|
| **3D в отдельном окне** | Tauri 2.0 поддерживает multi-window. Можно открыть 3D-вьюпорт в нативном окне с WebGL |
| **Физика** | Rust-библиотека `rapier` (WebAssembly из браузера, нативная из Tauri) — одна и та же библиотека! |
| **Генерация контента** | Rust в 10-50x быстрее JS для процедурной генереации карт, симуляции |
| **Доступ к GPU** | Через wgpu (Rust) можно рендерить в shared texture и отдавать в WebView — почти нативная производительность |
| **Steamworks** | Rust-биндинги к Steam SDK — лобби, друзья, Workshop, достижения |
| **Размер билда** | ~30 МБ против 150+ МБ у Electron |

### Что НЕ даёт Tauri для 3D (мифы)

- ❌ Tauri сам по себе не включает 3D-движок. Это просто WebView.
- ❌ WebView рендерит 3D так же как браузер (WebGL/WebGPU те же).
- ❌ Для НАСТОЯЩЕГО нативного 3D (Vulkan/DirectX) нужен дополнительный рендер-процесс на Rust.

## 🔥 ВАРИАНТ 3: Bevy (Rust Game Engine) + Tauri

Самый мощный, но сложный путь:

```
┌──────────────────────────────────────────┐
│              Tauri App                   │
│  ┌────────────┐    ┌──────────────────┐  │
│  │  WebView   │    │  Bevy Render     │  │
│  │  (UI, 2D)  │    │  Window (3D!)    │  │
│  │            │    │                  │  │
│  │ HTML/CSS   │    │ Vulkan/DirectX   │  │
│  │ React      │    │ Нативный рендер  │  │
│  └────────────┘    └──────────────────┘  │
│         ↕ IPC (сообщения) ↕              │
│  ┌─────────────────────────────────────┐ │
│  │        Rust Backend                 │ │
│  │  • rapier (физика)                  │ │
│  │  • Networking (WebSocket)           │ │
│  │  • File System                      │ │
│  │  • Steam SDK                        │ │
│  └─────────────────────────────────────┘ │
└──────────────────────────────────────────┘
```

**Bevy** — это ECS-движок на Rust:
- Vulkan/DirectX/Metal из коробки
- PBR-шейдеры, тени, bloom, HDR
- Встроенная ECS (как в твоём дизайн-документе!)
- Физика через rapier

### Плюсы
- 🟢 НАТИВНАЯ 3D-производительность (как у AAA-игр, не шучу)
- 🟢 Rust безопасность памяти без garbage collector'а
- 🟢 ECS архитектура = естественное продолжение entity-системы
- 🟢 Кроссплатформа: Windows, Mac, Linux

### Минусы
- 🔴 Огромный объём работы (Bevy — это game engine, не UI-фреймворк)
- 🔴 UI в Bevy примитивный. Весь React-UI будет в WebView, 3D в Bevy-окне
- 🔴 Два разных стека для UI и 3D = двойная работа
- 🔴 Bevy ещё не v1.0 (но уже production-ready для инди)

## 🆚 ВАРИАНТ 4: Godot

- 🟢 Полноценный 2D/3D движок со встроенным редактором
- 🟢 Мультиплеер из коробки (ENet, WebSocket)
- 🟢 GDScript простой, C# для сложной логики
- 🔴 UI система ограничена по сравнению с HTML/CSS
- 🔴 Нельзя просто взять React-код и перенести
- 🔴 Steam Workshop сложнее (нет файловой модели)

## 🧭 МОЯ РЕКОМЕНДАЦИЯ (ПРАГМАТИЧНАЯ)

### Сейчас (Версии 0.1 — 0.5): React + TS + Express
Ничего не менять. Фундамент правильный. Чиним мультиплеер.

### Версия 0.5+: Обернуть в Tauri
```
План миграции на Tauri:
1. Инициализировать Tauri-проект рядом с текущим
2. Перенести Express-логику в Rust (команды Tauri)
3. WebSocket сервер тоже в Rust
4. React-фронт остаётся БЕЗ ИЗМЕНЕНИЙ (WebView)
5. Билд = один .exe файл
```

Затраты: ~2-3 недели на миграцию.
Выгода: десктопное приложение, Steam-ready, быстрее сервер.

### Версия 0.7+: 3D-режим через Three.js в WebView
```
Три подхода на выбор (по возрастанию сложности):

ЛЁГКИЙ (Three.js в React):
├── canvas с WebGL внутри React-компонента
├── 3D-кости, миниатюры, стол
├── Физика через rapier WASM
├── Затраты: ~2-3 недели
└── Достаточно для 80% "Tabletop Simulator" опыта

СРЕДНИЙ (Tauri multi-window + Three.js):
├── 3D-сцена в отдельном окне Tauri
├── IPC между UI-окном и 3D-окном
├── Затраты: ~1 месяц
└── Лучшая производительность (нет UI-оверхеда)

ХАРДКОР (Tauri + Bevy):
├── Нативный 3D-движок в отдельном процессе
├── Vulkan/DirectX, PBR, полная физика
├── Затраты: ~3-6 месяцев
├── Результат: AAA-качество графики
└── Оправдано ТОЛЬКО если 3D станет ядром продукта
```

### Финальный план развития

```
Версия 0.1 ──── Починить мультиплеер, базовый геймплей
                  (React + TS + Express + Yjs)
                     │
Версия 0.5 ──── Tauri-обёртка, Steam-лобби
                  (React + TS + Tauri + Rust backend)
                     │
Версия 0.7 ──── 3D-кости, виртуальный стол
                  (Three.js в React, rapier WASM)
                     │
Версия 0.9 ──── Steam Workshop, ачивки
                  (Steamworks SDK через Tauri)
                     │
Версия 1.0 ──── Релиз в Steam
                     │
Версия 1.5+ ─── Если 3D взлетит → Bevy интеграция
                  (опционально, если надо AAA-графику)
```

## 💡 КЛЮЧЕВОЙ ВЫВОД

**Tauri + Rust имеет смысл**, но не для 3D как такового (Three.js в браузере даст 90% того же), а для:

1. **Десктопного билда** (один .exe, не надо браузер)
2. **Steamworks интеграции** (лобби, друзья, Workshop — без Hamachi!)
3. **Более быстрого сервера** (Rust вместо Express/Node)
4. **Нативного доступа к файловой системе** (быстрее загрузка ассетов)
5. **Меньшего размера** (30 МБ против 150+)

**Но начинать миграцию на Tauri надо НЕ сейчас**, а после версии 0.5, когда основной функционал отлажен в браузере.

**3D-режим НЕ требует менять стек.** Three.js в WebView — достаточно для виртуального стола с миниатюрами. Если трёхмерка станет ядром продукта — тогда Bevy. Но это решение на 2027 год, не сейчас.
