MASTER DESIGN DOCUMENT: Vibe TTRPG Platform
1. Концепция и Роль ИИ
Контекст для ИИ: Пользователь не является программистом и выступает в роли Product Manager'а. Твоя задача (ИИ) — писать надежный, модульный, легко читаемый код, строго следуя этой спецификации.
Цель проекта: Гибридная локальная VTT-платформа (Miro + Obsidian + FoundryVTT), построенная на идее Универсальных Сущностей.
Оптимизация Архитектуры (Важно для ИИ)
Пользователь описал 9 типов сущностей. Для оптимизации базы данных и упрощения логики мы применяем паттерн, близкий к ECS (Entity-Component-System).
Мы НЕ создаем 9 разных таблиц. У нас есть ЕДИНАЯ структура данных Entity, которая меняет свое поведение в зависимости от поля type и вложенных метаданных.
Разделяй Данные (Data) и Отображение (View). "Блоки механик" и "Блоки хранилищ" — это не столько отдельные данные, сколько UI-компоненты (виджеты), которые по-разному рендерят данные внутри Entity.
2. Технологический стек (Обязательный)
Core: React + TypeScript + Vite.
State & Multiplayer: Yjs (CRDT) + y-websocket (Топология "Звезда"). В экосистеме LAN (Hamachi/Radmin) y-websocket — это наиболее оптимизированное решение. Хост поднимает локальный WebSocket-сервер вместе с Express (без сложного внешнего бэкенда). Клиенты соединяются только с Хостом. Это избавляет от `O(N^2)` P2P-соединений y-webrtc, снижает задержки и не требует внешних сигнальных серверов. Эфемерные данные (Курсоры мыши, Указка) передаются через Yjs Awareness Protocol с ограничением кадров (throttling), чтобы не забивать сеть.
Local-First подход и Управление Памятью (Asset Management):
Платформа концептуально работает полностью локально (Local-First). ГМ может создать в офлайне весь мир, а затем при надобности "поднять сервер" и раздать ID комнаты.
ОПТИМИЗАЦИЯ ПАМЯТИ: Yjs синхронизирует ТОЛЬКО логические JSON-сущности (тексты, связи, метаданные, координаты). Тяжеловесные файлы (Картинки для Карт, Изображения, PDF, и в перспективе 3D Модели) НИКОГДА не сохраняются в Yjs в виде Base64-кода, чтобы не убить оперативную память (RAM). `imageId` является либо внешним URL (Imgur и т.д.), либо указателем на файл, который (как в Tabletop Simulator) в реальном времени скачивается в кэш браузера от хоста по мере того, как игрок переходит на этот экран.
UI & Windows: Tailwind CSS + Zustand (для локального состояния окон). Дизайн-система строго следует эстетике "Apple Glassmorphism" — матовые панели (bg-white/5, backdrop-blur-3xl), фокус окон без неоновых обводок (через интенсивность блюра), iOS-подобные сглаженные переключатели (toggle switches) и инпуты без тяжелых рамок.
Canvas Engine: react-konva. Оптимизировано для большого количества объектов.
Markdown: react-markdown с кастомными плагинами для вики-ссылок.
3. Универсальная Модель Данных (Unified Entity System)
Любой объект в системе описывается интерфейсом Entity.
code
TypeScript
type EntityType = 'character' | 'object' | 'ability' | 'tag' | 'canvas' | 'portal' | 'note' | 'folder';

interface Entity {
  id: string;
  parentId: string | null; // Для иерархии (папки, инвентарь)
  type: EntityType;
  name: string; // Имя должно быть глобально уникальным в базе
  description: string; // Markdown текст
  imageId?: string; // Заглушка или аватар для режима "Иконка"
  
  // Гибкая структура для характеристик. 
  // Пример: { strength: { base: 10 }, weight: { base: 5 } }
  properties: Record<string, any>; 
  
  // ID прикрепленных тегов
  tags: string[]; 
}
Реализация идей пользователя:
Сущность-Персонаж: Entity с типом character. Выступает одновременно и как папка, и как игровой объект. Хранит в себе ссылки на инвентарь, заклинания, оружия, черты (все они — вложенные сущности). В интерфейсе (левой и правой панели) персонажи визуально раскрываются как папки-аккордеоны, позволяя увидеть всё их внутреннее содержимое. Перемещение: из внешней базы (Датабазы) элементы копируются. Из инвентаря другого персонажа — появляется запрос "Скопировать или перенести?". Персонажа нельзя скопировать внутрь другого персонажа. То же самое касается перетаскивания атак (attack) внутрь предметов (object) - всегда должен появляться диалог "Скопировать или перенести", чтобы дать возможность переиспользовать шаблоны атак из генеральной базы.
Многоуровневая вложенность (Матрешка): Любая сущность (например, объект-оружие) также может выступать как папка. Пример: внутри папки Персонажа лежит обьект "Меч" (weapon), а внутри "Меча" лежат сущности 2-х разных атак ("Рубящая", "Колющая").
Агрегация в окне (UI Requirements): Для удобства игрока и ГМа, когда мы открываем Окно Персонажа, движок должен рекурсивно найти все атаки во всех вложенных в него объектах (мечах, луках) и показать их единым списком во вкладке "Атаки". Игроку не нужно проваливаться в папку меча, чтобы ударить мечом. При этом, если открыть отдельное Окно "Меча", внутри него также будет показан блок с его собственными атаками.
Сущность-Объект: Выступает как шаблон. При добавлении Объекта в Персонажа из БД происходит Deep Clone (создается новая сущность с parentId равным ID Персонажа).
Сущность-Заметка: Entity с типом note, где поле description рендерится через Markdown с парсингом ссылок вида [[имя]].

Уникальность Имен и Ссылки (ОБНОВЛЕНО):
Система разделяет сущности на три базы данных с разными правилами идентификации:
- **Генеральная База (General DB)**: Мастер-каталог всех сущностей мира. Имя сущности = уникальный идентификатор. Дубликаты имён запрещены. Wiki-ссылки `[[Имя]]` ищут ТОЛЬКО в этой базе.
- **Базы Игроков (User DB)**: Личные инвентари каждого игрока. Сущности идентифицируются по внутреннему `uid` (не по имени). Дубликаты имён разрешены (может быть 100 "Экскалибуров"). Ссылки на конкретные экземпляры через `[[#uid:...]]`.
- **База ГМ-а (GM DB)**: Личное хранилище ГМ-а. Правила как у User DB, но видимость — только для ГМ-а.

Поиск по ссылкам `[[Имя]]` — регистронезависимый (`[[телосложение]]` найдет `Телосложение`).

3.5. Файловое Хранение Сущностей (Entity File System) — РЕАЛИЗОВАНО

КЛЮЧЕВОЙ ПРИНЦИП: Каждая сущность — это `.md` файл с YAML-frontmatter. Файлы можно открывать в Obsidian, VS Code или любом текстовом редакторе. Можно закинуть `.md` файл в нужную папку — он станет сущностью в приложении.

Архитектура: Локальный Node.js сервер (Express, порт 3001) запускается параллельно с Vite dev-сервером через `start.bat`. Только хост (ГМ) запускает файловый сервер; игроки получают данные через Yjs/WebRTC.

Реализованные серверные модули (`server/src/`):
- `index.ts` — Express API сервер + WebSocket для уведомлений об изменениях файлов
- `worldManager.ts` — создание/открытие/валидация мира, управление структурой папок
- `fileManager.ts` — CRUD операции с .md файлами, парсинг/сериализация YAML, Матрёшка-вложенность, sanitizeFilename (лимит 50 символов для Windows path)
- `fileWatcher.ts` — отслеживание внешних изменений через chokidar, WebSocket broadcast, защита от sync loops через `recentWrites` Set
- `renameManager.ts` — атомарное каскадное переименование (create new → update all `[[wiki-links]]`, tags, source refs → delete old)

Реализованные клиентские модули (`app/src/services/`):
- `fileApi.ts` — API клиент с проверкой isHost, CRUD, WebSocket connection, auto-reconnect
- `fileSyncService.ts` — мост Файлы ↔ Yjs: загрузка мира → Yjs, debounced writeback (2 сек), обработка внешних изменений

API эндпоинты файлового сервера:
- `POST /api/world/create` и `/api/world/open` — управление миром
- `GET/POST/PUT/DELETE /api/entities` — CRUD сущностей
- `POST /api/entities/:id/rename` — атомарное переименование с каскадным обновлением ссылок
- `POST /api/entities/import` — импорт произвольного .md файла
- `GET /api/assets` — получение списка всех доступных изображений в локальном хранилище
- `POST /api/assets/upload` — загрузка новых изображений (сохраняются как файлы в папку `/assets/`, дублирующиеся имена файлов авто-инкрементируются для защиты от перезаписи)
- `GET /api/assets/:filename` — раздача медиа-файлов
- `WS /ws/watch` — real-time уведомления об изменениях файлов

Формат файла сущности:

```markdown
---
type: character
tags: [воин, игрок]
image: warrior.png
stats: { strength: 18, dexterity: 14 }
resources:
  hp: { max: "8 + $constitution", current: 32 }
properties:
  power:
    astral: { base: 3 }
---

# Торин Железнобокий

Ветеран северных войн, мастер двуручного меча.
```

Правила frontmatter:
- `type` — обязательно, определяет тип и папку хранения
- `tags` — массив тегов (inline: `[тег1, тег2]`)
- `stats` — характеристики персонажа (сокращённо: `strength: 18` → в Yjs `{ base: 18 }`)
- `resources` — ресурсы с `current`/`max` (HP, мана)
- `properties` — произвольные данные (используется для type != character и для дополнительных полей)
- `image` — ID ассета (имя файла в `assets/`)
- `# Заголовок` — первый H1 в body = имя сущности
- Остальной body = description

Система вложенности «Матрёшка» (РЕАЛИЗОВАНО):
Сущности содержат другие сущности через файловую систему. Рядом с `.md` файлом создаётся папка с таким же именем:

```
general/characters/
├── Торин Железнобокий.md           ← Сам персонаж
└── Торин Железнобокий/             ← Папка = его содержимое
    ├── Экскалибур.md               ← Меч (parentId: "Торин Железнобокий")
    └── Экскалибур/                 ← Содержимое меча
        └── Рубящая Атака.md        ← Атака (parentId: "Экскалибур")
```

`parentId` вычисляется автоматически из пути файла при загрузке — не хранится в frontmatter.

Структура папок мира:

```
/world/
├── world.yaml                 # Метаданные мира (name, createdAt)
├── general/                   # 🏛️ ГЕНЕРАЛЬНАЯ БАЗА (имя = ID)
│   ├── characters/            # Персонажи
│   ├── objects/               # Предметы
│   ├── abilities/             # Способности
│   ├── tags/                  # Теги (с подпапками: hidden/, properties/, statuses/)
│   ├── notes/                 # Заметки
│   └── canvases/              # Пространства
├── users/                     # 👤 БАЗЫ ИГРОКОВ (uid = ID)
│   ├── player_Alex/           # Инвентарь Alex
│   └── player_Maria/          # Инвентарь Maria
├── gm/                        # 🎭 БАЗА ГМ-а
└── assets/                    # 🖼️ Медиа-файлы
```

Правила идентификации по базам:

| База | Имя файла | Идентификатор | Дубликаты | Видимость |
|------|-----------|---------------|-----------|-----------|
| General | `{Имя}.md` | Имя файла | ❌ | Все |
| Users | `{Имя}.md` | `uid` в YAML | ✅ | Владелец + ГМ |
| GM | `{Имя}.md` | `uid` в YAML | ✅ | Только ГМ |

Кастомные расширения Markdown (специфичные для Vibe TTRPG):

- `[[Экскалибур]]` → ссылка на сущность в General базе
- `[[#uid:inv_001]]` → ссылка на конкретный экземпляр по uid
- `` `!roll 2d6+4` `` → интерактивная кнопка броска кубов
- `` `!roll 1d20+$strength` `` → бросок с подставкой стата
- ` ```inventory ``` ` → интерактивный блок инвентаря с drag & drop
- ` ```stats ``` ` → визуальный блок статов (HP бары)
- ` ```gm-only ``` ` → блок видимый только ГМ-у

Гибридная архитектура (файлы + мультиплеер) — РЕАЛИЗОВАНО:

- Файлы на диске = источник истины (долговременное хранение)
- Yjs (y-websocket) = транспорт для реал-тайм синхронизации между игроками в формате "Клиент-Сервер (Хост)"
- Раздельные Y.Docs (Subdocuments): Канвасы грузятся в отдельные WebSocket каналы (комнаты), чтобы не засорять общую память при активном рисовании.
- Защита от разрастания CRDT (Tombstones): Из-за постоянного рисования Yjs накапливает удаленные "мусорные" операции. Раз в сессию Yjs документ должен быть сброшен путем принудительной переинициализации из файловой системы.
- При запуске: хост загружает файлы → парсит → заполняет Yjs → игроки получают данные (fileSyncService.ts)
- Изменения через Yjs записываются обратно в файлы (debounce 2 сек, только у хоста)
- IndexedDB (y-indexeddb) = кэш для быстрого старта, не источник истины
- Внешние изменения (Obsidian/VS Code) → chokidar → WebSocket → Yjs → все игроки
- Sync loop защита: `recentWrites` Set + `_isLoading` flag

Система Папок (Folders):
Тип `folder` используется для визуальной и логической группировки. В Датабазе (Hub) и Личном Инвентаре сущности могут вкладываться в папки. Обязательные 3 папки внутри группы "Теги" для организации механик: "Скрытые теги", "Статусы", "Свойства".
4. Математический Движок (Modifier Engine)
Система динамического подсчета характеристик (Сила, Урон и т.д.).
Это ядро логики платформы. Характеристика не является просто числом, она вычисляется на лету.
Формула: Total = Base Value + Modifiers (from Tags) + Modifiers (from Mechanic Blocks) + AdHoc (Ручной ввод)

Гибкость зависимостей (Graph of Dependencies): 
Система вычислений должна позволять любой характеристике влиять на любую другую характеристику. Например, способность "Атлет" дает +2 к "Телосложению", а "Телосложение" вместе с базовой "Фигурой" формируют "Переносимый вес" (Total = Base + CON + Size). В интерфейсе (Окне свойств) у пользователя должна быть возможность гибко привязывать модификаторы способностей к любым статам без необходимости лезть в код.

- Наследование параметров (Context Bubbling):
Так как у нас внедрена система многоуровневой "Матрешки" (Персонаж -> Меч -> Атака Мечом), формулы должны уметь подниматься вверх по дереву `parentId` в поисках нужной характеристики. Если Атака требует стат "Мощь" (Might), но внутри самой сущности Атаки его нет, математический движок ищет его в Оружии, а затем — в Персонаже. Аналогично, свойства родителей могут спускаться вниз (например, Персонаж дает бонус "Ловкости" всем своим кинжалам).
Техническое требование к ИИ:
Создать функцию-хук useCalculatedStat(entityId, statName). Она должна:
Брать базовое значение из объекта.
Проверять все tags объекта. Если у тега есть свойство modifiers, применять его.
Отдавать объект: { total: number, breakdown: Array<{source: string, value: number}> }.
UI Требование: При наведении на любое числовое значение в игре (Урон, Здоровье) выводить Tooltip с breakdown (например: "База: 3, Тег 'Разрез': +3 = Итого: 6"). У каждой характеристики в UI должна быть возможность явного редактирования как "Базового значения", так и "Дополнительного модификатора" (ручного AdHoc значения).

Конкретные Блоки Характеристик Персонажа:
- Блок "Атрибуты": Базовые параметры уровня существа (Телосложение, Когниция, Фигура, Мышление, Скорость, Голод). Выступают фундаментом для всех остальных расчетов в игре.
- Блок "Мощь" (Важнейший выделяющийся блок): Включает три вида энергии: Мощь Астрала, Мощь Эфира, Мощь Ауры. Блок требует UI-селектора "Текущего источника мощи". Выбранный источник переопределяет итоговую характеристику "Мощь", которая затем идет в формулы урона заклинаний или уклонения. Поддерживает двойное/тройное использование энергий (с выбором приоритета при двойном).
- Блок "Защита": Включает "Уклонение" и "Защиту". Уклонение зависит от модификатора Мощи по формуле: `math.floor(Мощь / 2)`.

Глобальные UI/UX Требования к Взаимодействию:
- Редактирование Сущностей: Во Full окнах названия сущностей и их описания должны быть редактируемыми. Описание (Description) всегда парсит Markdown и вики-ссылки (поведение заметки). 
- Языковое разделение: **Язык интерфейса** — **Русский** (для игрока: названия статов, тултипы, меню, "Расчёт модификаторов", инвентарь, Хаб датабазы). **Язык кода, переменных, типов и системных сущностей** — **Английский** (Entity, Object, Tag, Canvas, hp, wounds, base, modifiers, properties, типы тегов и т.д).
- Переименование и другие действия: Должно быть реализовано глобальное Контекстное меню (вызов по ПКМ) для порталов, сущностей в базе/инвентаре и в окне.
  - **Дизайн контекстного меню**: Должно появляться строго под курсором мыши, не вылезая за края экрана. Все контекстные меню в приложении (Порталы на Канвасе, База данных, Шапка окна) **обязаны быть визуально унифицированы**. Используется единый класс стеклянного контейнера (`bg-[#151c2b]/70 backdrop-blur-3xl border border-white/10 rounded-xl`), единый заголовок капсом (`tracking-widest text-[9px] border-b border-white/5`) и идентичный набор отступов с иконками Lucide размера 14. Дублирование разных стилей контекстного меню запрещено.
  - **Перемещение (Drag & Drop Prompt)**: При переносе сущности из базы в инвентарь, вместо браузерного `window.confirm`, появляется кастомный красивый поповер (Pop-up) с выбором: Перенести, Копировать, Отмена.
- Удаление сущностей: **Каждое удаление любой сущности требует явного подтверждения (Confirmation Dialog)**. Это касается кнопок корзины везде в интерфейсе (В инвентаре, в базах).
- Удаление в режиме отладки: **В режиме отладки (Debug Mode) должна присутствовать глобальная кнопка для удаления открытой сущности**.
- Автоматическое связывание с Заметками (Wiki-Linking): Каждое название блока (например, "Атрибуты") и каждое название характеристики (например, "Телосложение") в UI должно работать как гиперссылка. При нажатии ЛКМ на название характеристики система должна искать сущность типа `note` (Заметка) с таким же названием (ОБЯЗАТЕЛЬНО БЕЗ УЧЕТА РЕГИСТРА). Если она есть — открывать её окно, давая игрокам мгновенный доступ к правилам.

Категории Тегов (Tags Categories):
- Скрытые теги: Технические метки (например "оружие", "броня"), скрыты от глаз обычных игроков. Нужны логике, чтобы понимать класс объекта (Свойство "отражение" вешается только на броню). Отображаются только в Режиме Отладки.
- Статусы: Теги, которые могут быть навешены только на персонажей ("Оглушон", "Поломан"). Могут иметь как просто нарративный характер ("Безмолвие"), так и математический модификатор к статам. Отображаются в публичном блоке "Статусы" у 캐릭터а. Поддерживают механику Рангов (Например: Слепота 1 уровня снижает уклонение на -1, Слепота 2 уровня на -2).
- Свойства: Теги для игровых предметов (оружия, брони, и тд). Могут менять статы предмета или персонажа, надевшего предмет. Отображаются в блоке "Свойства" в окне предмета. Также обладают системой Рангов. Скрытые объекты не имеют блока тегов в обычном UI.
5. Система Окон и 3 Режима Отображения (Window Manager)

Визуальная структура Окна (EntityWindow):
Каждое окно сущности имеет сверху универсальный визуальный блок `EntityImageBlock`. 
- Для Канваса (canvas) фото имеет широкий альбомный формат (для пейзажей).
- Для остальных сущностей (character, object, attack и т.д.) — квадратный.
- При отсутствии фото выводится полупрозрачная иконка-заглушка в зависимости от типа сущности (Меч, Коробка, Человек и т.д.).
- При наведении на блок (для ГМа/Host-а) открывается стеклянная панель управления фото: можно загрузить новый файл с компьютера (уйдет на `API /assets/upload`), выбрать ранее загруженный файл из выпадающего списка (чтобы не дублировать память сервера) или вставить прямую URL-ссылку из интернета. Удаление фото из сущности стирает лишь текстовую ссылку в БД, сохраняя исходный файл на жестком диске.

Для любой Entity пользователь может открыть окно. У окна есть 3 состояния (View Modes):
Debug (Режим отладки): Аналог бывшего полного режима (Full) — позволяет ГМу и разработчику видеть все скрытые свойства, скрытые теги, сырые данные (JSON properties) для тюнинга сущности. Иконка — "жук".
Compact (Малое): Уплотненный UI, скрыты вторичные данные (например, только ХП и атаки).
Icon (Шарик/Иконка): Маленький круг с аватаркой. При двойном клике разворачивается.
Оптимизация "Закрепления на Канвасе":
Если окно не закреплено, оно "плавает" поверх всего интерфейса (HTML-слой). Если пользователь нажимает "Pin to Canvas", окно становится узлом (Node) внутри react-konva и получает координаты x, y на текущем холсте.

Иерархия Слоев (Z-Index Hierarchy):
Для правильного позиционирования UI используется строгий порядок слоев:
1. Активный Канвас (z:0): Самый нижний слой для рендера графики (сетка, рисунки).
2. Элементы Канваса (z:10): Закрепленные окна, порталы, прикрепленные заметки. Масштабируются вместе с зумом канваса.
3. Активный UI Канваса (z:20): Кнопка рецентра (в правом нижнем углу), список "Активных элементов", будущая панель инструментов (toolbar). Выводятся поверх элементов канваса.
4. Кнопки вызова боковых панелей (z:30): Кнопки для открытия базы (Hub) и инвентаря игрока.
5. Боковые панели (Drawers) (z:40): Выдвижные панели (Левый Инвентарь, Правая Датабаза/Чат) и всё их содержимое.
6. Незакрепленные окна (z:50): Окна и их миниатюры всегда отображаются выше боковых панелей, чтобы перекрывать открытые менюшки слева и справа.
7. Всплывашки (Tooltips, Popovers) (z:9999): Всплывающие подсказки и поповеры, они рендерятся как React Portal в корне body поверх абсолютно всего приложения.
- **Анимации Tooltip / Popover**: Избегаем тяжелых transition/animations из Tailwind (например, убираем `animate-in zoom-in fade-in` у `FloatingPortal` в пользу мгновенного появления `z-[9999]`), чтобы избежать глитчей позиционирования при быстрых кликах.

6. Система Бесконечных Канвасов (Canvas Routing)
Оптимизация архитектуры:
Мы НЕ рисуем холст внутри холста (это убьет производительность). Мы используем принцип доменной маршрутизации (Routing).
Есть словарь всех холстов (сохраненных как Entity типа canvas).
Есть GlobalStore, который хранит activeCanvasId.
На холсте "Страна" пользователь создает "Портал" (Картинка города). Этому порталу присваивается ссылка на activeCanvasId = ID_Города.
При двойном клике на портал, экран мгновенно очищается и рендерит объекты из массива холста "Город". Добавляется кнопка "Назад" (хлебные крошки).
Контекстное меню Порталов: При клике правой кнопкой мыши по Порталу открывается стеклянное контекстное меню с возможностью "Открыть окно области". Это открывает стандартное EntityWindow Канваса, куда ведет портал, позволяя игрокам прочитать текст описания и посмотреть панорамную фотографию `EntityImageBlock` этой локации без непосредственного перемещения пространства.
7. Чат, Дайсы (Кости), Мультиплеер и UI Лейаут
Мультиплеер: Реализуется через управляемый Yjs WebSocket. Хост запускает сервер. Игроки подключаются по прямому IP-адресу (LAN IP). Все действия (перемещение иконок, изменение ХП) мгновенно синхронизируются у всех через хаб (Хоста). Для оптимизации требуется группировка (batching) частых событий (Drag-n-Drop объектов). Эфемерные данные (Указка, Курсоры) синхронизируются через Yjs Awareness Protocol без сохранения в CRDT историю (с Throttling'ом в ~30fps).
Чат: Отдельная панель (вкладка в правом меню). Хранится как Y.Array сообщений.
Дайсы: Если пользователь в чате пишет /r 2d6 или нажимает кнопку в "Блоке Механики Навыков", генератор случайных чисел выдает результат и отправляет его в Чат с пометкой.

7.5 Панели Интерфейса (Левое и Правое меню)
Правое выдвижное меню (База и Коммуникация):
Включает в себя вкладки:
- База данных: хранилище всех сущностей и заметок. Все разбито по категориям.
- Чат: чат всех подключенных игроков, история бросков кубиков и системных уведомлений.
Левое выдвижное меню (Личный Инвентарь):
- Личное хранилище (инвентарь сущностей) игрока или ГМ-а. У каждого участника оно личное.
- В будущем (при реализации мультиплеера) ГМ сможет переключать вкладки для просмотра инвентаря конкретных игроков.
