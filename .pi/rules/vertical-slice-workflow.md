# 🍰 Vertical Slice Workflow

> Как мы работаем в vibe-стиле. От вайба — до кода.

---

## 1. ФИЛОСОФИЯ

**Vibe Coding** = итеративная разработка полными вертикальными срезами.

Вместо «сделаем всю БД, потом весь бекенд, потом весь фронтенд» — делаем одну фичу **сразу через все слои**:
```
DB → API → Store → UI/Canvas → Tests → Docs
```

Почему:
- Каждый слайс — демонстрируемый результат
- Риск блокировки минимален (никакой слой не ждёт другой)
- Быстрая обратная связь (видно сразу)
- Легко менять направление

## 2. ЦИКЛ РАБОТЫ

```
┌──────────────────────────────────────────────────┐
│  1. ВАЙБ  — Что хотим? Какая фича?             │
│      ↓                                           │
│  2. PRD   — Mini-PRD (полстраницы)              │
│      ↓                                           │
│  3. ПЛАН  — Вертикальные слайсы                │
│      ↓                                           │
│  4. КОД   — Итеративная реализация              │
│      ↓                                           │
│  5. КРИТИКА — Self-critique, edge cases         │
│      ↓                                           │
│  6. ТЕСТЫ — Модульные + интеграционные          │
│      ↓                                           │
│  7. ДОКИ  — .pi/docs/фича.md                   │
└──────────────────────────────────────────────────┘
```

---

## 3. ШАГ 1: ВАЙБ (Vibe)

Формат: свободный текст. Что хотим, какую проблему решаем.

```
Пример:
"Хочу, чтобы ГМ мог выдавать предметы игрокам drag&drop из общей базы в их инвентарь.
Сейчас предметы можно только вручную создавать в user-базе."
```

### Уточняющие вопросы:
- Кто пользователь? (ГМ, игрок, все)
- Где в UI это происходит?
- Какие ограничения? (только определённые типы? только между базами?)
- Что будет если ошибка? (предмет исчез? дубликат?)

Шаблон (полстраницы, не больше):

```markdown
## Проблема
[1-2 предложения]

## Решение
[2-3 предложения]

## User Story
- Как [роль], я хочу [действие], чтобы [ценность]

## Acceptance Criteria
- [ ] Критерий 1
- [ ] Критерий 2
- [ ] Критерий 3

## Edge Cases
- Случай 1: [описание] → [ожидаемое поведение]
- Случай 2: [описание] → [ожидаемое поведение]

## Технические заметки
- Затрагивает файлы: [список]
- Новые зависимости: [нет / список]
- Риски: [список]
```

## 5. ШАГ 3: ПЛАН ВЕРТИКАЛЬНЫМИ СЛАЙСАМИ

```markdown
## Вертикальные слайсы

### Слайс 1: [название] (end-to-end проход)
- DB: [что в .md / что в Yjs]
- Server: [новые endpoint'ы]
- Store: [новые/изменяемые Zustand сторы]
- UI: [компоненты]
- Math: [если нужны вычисления]
- Tests: [что тестируем]

### Слайс 2: [название]
...
```

### Принципы слайсов:
1. Каждый слайс — **работающий** инкремент (можно показать)
2. Слайс 1 — самый тонкий (MVP фичи)
3. Слайс 2+ — дополнения, полировка, edge cases
4. Если слайс > 200 строк кода — разбей на под-слайсы

---

## 6. ШАГ 4: КОД (Итеративно)

### Порядок в каждом слайсе:
1. **Типы** — обнови интерфейсы
2. **Стор/Сервер** — данные и логика
3. **UI** — компоненты
4. **Стили** — Tailwind через theme.ts
5. **Интеграция** — соедини с существующим

### Правила генерации:
- Показывай код файла целиком (write) или diff (edit)
- После каждого файла — краткий комментарий что изменилось
- Не генерируй >3 файлов за раз без подтверждения

---

## 7. ШАГ 5: SELF-CRITIQUE

После каждого слайса — проверь:

### Чеклист
```
☐ Работает happy path
☐ Обработаны error states
☐ Нет утечек памяти (useEffect cleanup)
☐ Нет лишних ререндеров (useMemo/useCallback где надо)
☐ Совместимость с canvas (координаты, слои)
☐ Совместимость с Obsidian (.md формат)
☐ Мультиплеер (Yjs синхронизация не сломана)
☐ Права доступа (игрок не может то, что может только ГМ)
☐ Z-index не сломан
☐ Стили используют theme.ts (не хардкод)
```

### Edge Cases (TTRPG-специфичные):
```
☐ Что если стат не найден? (context bubbling)
☐ Что если сущность удалена во время редактирования?
☐ Что если два игрока одновременно двигают токен?
☐ Что если дайс-ролл с отрицательным модификатором?
☐ Что если canvas переключили во время drag&drop?
☐ Что если имя сущности уже занято?
☐ Что если .md файл вручную отредактирован с ошибкой YAML?
```

## 8. ШАГ 6: ТЕСТЫ

### Что тестируем всегда:
```typescript
// 1. Чистые функции (utils)
describe('diceParser', () => { ... });
describe('entitySerializer', () => { ... });

// 2. Хуки
describe('useCalculatedStat', () => { ... });

// 3. Сторы
describe('windowStore', () => { ... });
```

### Что тестируем иногда (сложные интеграции):
- Компоненты (React Testing Library)
- API endpoints (supertest)

### Формат тестов:
- Рядом с файлом: `diceParser.test.ts`
- Или в `__tests__/` папке

---

## 9. ШАГ 7: ДОКУМЕНТАЦИЯ

После завершения фичи:
1. Создай `.pi/docs/[фича].md`
2. Опиши: что сделано, архитектуру, как тестировать
3. Обнови AGENTS.md при необходимости
4. Обнови DEVELOPMENT_PLAN.md (отметь завершённое)

## 10. КОММУНИКАЦИЯ

### Агент → Пользователь:
```
"План:
 Слайс 1: EntitySelector в WindowManager
 Слайс 2: Drag&Drop логика
 Слайс 3: Контекстное меню

 Начинаю слайс 1?"
```

### После кода:
```
"Готово. Файлы: X.tsx, Y.ts. 
 Self-critique: нашёл 2 edge case'а — ...
 Применить? Доработать? Следующий слайс?"
```

### Caveman Mode:
Кратко. По делу. Без воды.
```
"Понял. План:
1. Добавить тип EntityLinkToken
2. Рендер в PortalLayer
3. Клик = navigate
→ Делаю."
```

## 11. ЗАПРЕТЫ

```
❌ Не делай больше одного слайса за раз без подтверждения
❌ Не меняй архитектуру без обсуждения
❌ Не добавляй зависимости без явного OK
❌ Не игнорируй мультиплеер (всегда думай: а что если два клиента?)
❌ Не удаляй существующий функционал без ConfirmDialog
❌ Не оставляй console.log в production-коде
❌ Не хардкодь строки — используй i18n
```

---

*Vibe coding = дисциплина, а не анархия. Каждый слайс — как спелый фрукт.*
