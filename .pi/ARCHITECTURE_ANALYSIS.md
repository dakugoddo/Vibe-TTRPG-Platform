> сотни документов, десяток игроков, десятки канвасов (от маленьких до гигантских боевых полей)

## 🔴 КРИТИЧЕСКИЕ ПРОБЛЕМЫ ТЕКУЩЕГО ФУНДАМЕНТА

### Проблема #1: CRDT для рисования = катастрофа памяти

Каждый штрих кисти, каждое перемещение токена сохраняется в Yjs CRDT-историю НАВСЕГДА (tombstones).

Сценарий: 20 токенов, 4 игрока, 30 минут боя:
```
Операций перемещения: 20 токенов × 4 игрока × ~10 движений/сек × 3600 сек = ~2.8 МИЛЛИОНА ОПЕРАЦИЙ
Размер Y.Doc в памяти: 200-500 МБ на одного клиента
```

"State Compaction" из FUTURE_DEVELOPMENT_PLAN.md (пересоздавать Y.Doc из файлов раз в сессию) — костыль. Не решает проблему накопления ВНУТРИ сессии.

**Правильное решение:**
- Рисование: НЕ через CRDT. Только финальный результат.
- Перемещение токенов (drag): ephemeral сообщения во время drag, CRDT только на drop.
- Изменение статов (HP): через CRDT (они редкие).

### Проблема #2: y-websocket НЕСОВМЕСТИМЫ

```
server/package.json: "y-websocket": "^1.5.0"
app/package.json:    "y-websocket": "^3.0.0"
```

V1 и V3 протоколы НЕСОВМЕСТИМЫ. Мультиплеер не работает вообще.

### Проблема #3: Нет эфемерного транспорта

Весь канвас идёт через `Y.Map<DrawElement>`:
- Каждый drag-move токена → полная запись в CRDT
- Каждый штрих пера → десятки операций в CRDT
- Awareness (курсоры) не настроен

### Проблема #4: react-konva не для тактических сцен

- Нет встроенного tilemap-рендеринга
- Нет spatial hashing для frustum culling
- Для тысяч объектов нужен кастомный подход

### Проблема #5: Всё сохраняется мгновенно

fileSyncService пишет ВСЕ изменения через debounce 2 сек. Нет разделения между "боевой порядок" (критично) и "каракули" (можно по кнопке).

## ✅ ЧТО ХОРОШО И НАДО СОХРАНИТЬ

| Компонент | Почему |
|-----------|--------|
| Yjs для entities | Сотни сущностей, редкие изменения = идеальный CRDT |
| Sub-documents | Отдельный Y.Doc на канвас, игроки не качают всё сразу |
| Star-топология | O(N) соединений вместо O(N²) |
| .md файлы | Source of truth, Obsidian-совместимость, Steam Workshop |
| Разделение БД | general/user/gm — правильная модель |
| Window Manager + Pin | Закреплённые окна на канвасе |
| React+TS+Vite | Не надо менять, TS идеален |
| Zustand | Лёгкие сторы без лишних ререндеров |

## 🏗️ ЦЕЛЕВАЯ АРХИТЕКТУРА ПОД МАСШТАБ

### Трёхуровневая синхронизация

```
Уровень 1: PERSISTENT (CRDT через Yjs)
├── Сущности (character, object, ability, note, tag)
├── Чат сообщения
├── Структура мира (какие канвасы существуют)
└── ФИНАЛЬНЫЕ позиции токенов (только после drop)

Уровень 2: EPHEMERAL (WebSocket бинарные сообщения)
├── Перемещение токенов во время drag (throttled 20fps)
├── Рисование (только финальный штрих, не каждая точка)
├── Пульсирующий пинг ГМа
└── Состояния "сейчас происходит бросок куба"

Уровень 3: AWARENESS (Yjs Awareness Protocol)
├── Курсоры мыши всех игроков
├── Выделенные объекты (что игрок сейчас выбрал)
└── Текущий viewport (куда игрок смотрит)
```

### Гибридное сохранение канвасов

```
В реальном времени (CRDT):
✅ Позиции токенов (финальные, после drag-drop)
✅ Статы (HP, статусы)
✅ Новые/удалённые рисунки
✅ Новые/удалённые закреплённые окна

По требованию (кнопка Save / авто-сейв):
🟡 Каракули рисования
🟡 Промежуточные позиции при drag
🟡 Промежуточные точки ломаной линии

Авто-сейв:
1. По кнопке "Save Canvas" (ручное)
2. При уходе с канваса (если isDirty)
3. Каждые 5 минут (если есть несохранённые изменения)
```

### Гибридный канвас-движок

```
react-konva (текущий):
├── Small/medium канвасы (заметки, карты городов, простые сцены)
├── Закреплённые окна
└── Инструменты рисования

PixiJS (будущий слой для тактики):
├── Тактические боевые сцены (20+ токенов, туман войны, эффекты)
├── Опциональный режим "Tactical Mode"
└── Отдельный WebSocket канал для real-time позиций
```

## 🎮 STEAM WORKSHOP СОВМЕСТИМОСТЬ

Текущая .md файловая система — идеальный формат:
```
Steam Workshop Item = папка мира
├── world.yaml          
├── general/            (общие .md файлы)
├── assets/             (картинки, музыка)
└── .index.json         
```

Игрок подписывается → Steam скачивает папку → игра открывает как мир.
